CLAUDE.txt

Claude-specific coding-agent operating manual for my-dev-kit-orchestrator.

Project identity

my-dev-kit-orchestrator is a CLI-first workflow tool for design-first software development with coding agents.

It is not:
- my-dev-kit
- my-dev-kit-lab
- a security-validation framework
- a fully autonomous multi-agent platform in v0.1.0

Claude should treat this repository as a workflow-shell project that generates stage-specific prompts and tracks text artifacts on disk.

v0.1.0 scope

The v0.1.0 product is intentionally small.

Command surface:
- my-dev-kit-orchestrator init
- my-dev-kit-orchestrator start
- my-dev-kit-orchestrator status
- my-dev-kit-orchestrator prompt
- my-dev-kit-orchestrator list

Supported modes:
- feature
- repair
- test
- refactor
- harden

Run storage:
- local run folders under .my-dev-kit-orchestrator/runs/
- plain text prompts and artifacts
- simple stage advancement based on expected artifact file existence

The CLI manages workflow order, prompt generation, and run state.
The coding agent executes one bounded stage at a time.

v0.1.0 non-goals

Do not expand v0.1.0 into:
- direct LLM execution
- full JSON schema validation
- automatic my-dev-kit execution
- design-map generation
- automatic judge routing
- extra low-level CLI commands beyond init, start, status, prompt, and list

my-dev-kit may be referenced in architecture-context prompts when available, but my-dev-kit-orchestrator does not execute it automatically in v0.1.0.

Claude operating rules

Before editing:
- read the relevant instruction and product files for the current task
- verify the implemented behavior instead of relying on planning docs alone
- use the existing architecture and command surface

While editing:
- prefer small, bounded changes
- do not create parallel architecture
- do not over-engineer
- do not invent future features
- do not convert planning or source documents into implementation claims
- do not broaden scope without stating why

When uncertain:
- report the exact missing context
- state what could not be verified
- keep uncertainty explicit instead of guessing

Prompt-generation discipline

Generated prompts must be stage-specific.

Do:
- generate one prompt for the current stage
- include only the required input artifacts
- require one named output artifact
- include stop conditions and missing-context rules

Do not:
- generate one giant master prompt
- include unrelated workflow modes in a stage prompt
- include reusable concrete scenario examples
- include implementation instructions before implementation stages
- include test-file writing instructions before test-implementation stages
- include verification claims before verification stages

Artifact discipline

Artifacts are plain text in v0.1.0.

Core rules:
- the PseudocodePacket is the shared source for implementation and test implementation
- the TestStrategyPacket is the source for test implementation
- verification reports must contain actual command evidence
- final reports must state unresolved risks honestly
- file existence advances the workflow, not unsupported claims about artifact quality

Git and file discipline

Do not:
- commit IDE metadata
- commit generated local workflow runs
- remove AGENTS.md or CLAUDE.md
- rename agents.txt unless explicitly instructed
- create duplicate instruction files with conflicting guidance

If AGENTS.md or CLAUDE.md are thin pointers, keep them thin.

Verification and reporting

Do not claim:
- tests passed unless commands were run
- typecheck passed unless commands were run
- build passed unless commands were run
- behavior changed unless the code or docs actually changed

Always report:
- commands run
- exit codes
- skipped checks and the reason they were skipped
- remaining risks or verification gaps

Documentation behavior

When working on docs or instruction files:
- keep text practical and repository-specific
- use exact command names and file paths
- separate implemented v0.1.0 behavior from future possibilities
- do not describe roadmap items as already available
