ARTIFACT_CONTRACTS.txt

Project name:
my-dev-kit-orchestrator

Document purpose:
This document defines the artifact contracts used by my-dev-kit-orchestrator.

Artifacts are the handoff mechanism between workflow stages.

The CLI should not rely on one giant prompt or on the coding agent remembering everything from previous stages. Each stage produces a named artifact. Later stages consume those artifacts as binding context.

The artifacts do not need to be strict JSON schemas in v0.1.0. Text files are acceptable. The important requirement is that each artifact has a clear purpose, expected content, and role in the workflow.

v0.1.0 should prioritize useful, readable, inspectable text artifacts over complex schema validation.


1. Core artifact principle

Artifacts are the source of continuity between stages.

The workflow should follow this handoff chain:

RequestBrief
→ ArchitectureContextPacket
→ BehaviorModel
→ PseudocodePacket
→ TestStrategyPacket
→ ImplementationReport
→ TestImplementationReport
→ VerificationReport
→ JudgeReport
→ FinalReport

Each artifact should answer:

- What stage produced this?
- What prior artifacts were used?
- What decision or understanding does this artifact preserve?
- What later stages depend on this artifact?
- What is still uncertain or unresolved?

Artifacts should be compact enough to be reused in prompts, but complete enough to avoid reinterpreting the task from scratch.


2. Artifact storage

Default run folder:

.my-dev-kit-orchestrator/runs/<run-id>/

Recommended artifact folder:

.my-dev-kit-orchestrator/runs/<run-id>/artifacts/

Recommended core artifact files:

artifacts/
  request-brief.txt
  architecture-context-packet.txt
  behavior-model.txt
  pseudocode-packet.txt
  test-strategy-packet.txt
  implementation-report.txt
  test-implementation-report.txt
  verification-report.txt
  judge-report.txt
  final-report.txt

Mode-specific artifacts may also be stored in the same artifacts folder.

Examples of mode-specific artifact filenames:

artifacts/
  observed-behavior-report.txt
  behavior-trace.txt
  divergence-report.txt
  correction-design.txt
  regression-test-strategy.txt
  test-target-brief.txt
  behavior-reconstruction.txt
  pseudocode-summary.txt
  refactor-brief.txt
  existing-behavior-map.txt
  preserved-invariant-list.txt
  compatibility-test-strategy.txt
  hardening-brief.txt
  assumption-report.txt
  failure-mode-matrix.txt

The CLI can use simple file-existence checks to determine whether a stage artifact is present.


3. Artifact format for v0.1.0

v0.1.0 artifacts should be plain text.

Preferred structure:

Artifact:
<artifact name>

Workflow mode:
<feature | repair | test | refactor | harden>

Stage:
<stage name>

Inputs used:
- <prior artifact or source>
- <prior artifact or source>

Summary:
<short summary>

Content:
<sectioned artifact body>

Open questions:
- <uncertainty or missing information>

Downstream use:
<how later stages should use this artifact>

Status:
<complete | incomplete | blocked>

Plain text keeps the project simple and makes artifacts easy to paste into coding-agent prompts.

Later versions may add optional JSON schemas, but v0.1.0 should not require them.


4. Core artifacts

4.1. RequestBrief

Produced by:
Request brief stage

Used by:
- Architecture context stage
- Behavior model stage
- Final report stage

Purpose:
Convert the raw user request into a compact software-change brief.

The RequestBrief defines what the user is asking for before project context, behavior design, pseudocode, or implementation begins.

Required content:
- original request
- workflow mode
- requested change
- target project or repository if known
- target workflow, component, module, command, or behavior if known
- user-visible or externally observable behavior
- constraints
- non-goals
- success criteria
- ambiguity or missing information
- expected next stage

Should not contain:
- implementation code
- test files
- detailed pseudocode
- speculative architecture changes
- broad unrelated project analysis

Completion criteria:
The RequestBrief is complete when a later stage can understand what kind of software change is being requested and what success means.

4.2. ArchitectureContextPacket

Produced by:
Architecture context stage

Used by:
- Behavior model stage
- Pseudocode packet stage
- Test strategy stage
- Implementation stage
- Judge stage

Purpose:
Ground the workflow in the current project architecture before design or implementation decisions are made.

The ArchitectureContextPacket should be created through focused inspection, preferably using my-dev-kit and graph-guided retrieval when available.

Required content:

- relevant tests

- likely files or modules involved
- context gaps
- reasons each major file or symbol was selected

- implementation changes
- final behavior design
- test strategy
- speculative rewrites
- broad unrelated file listings

Completion criteria:
The ArchitectureContextPacket is complete when the next stage can design behavior using current project evidence instead of guessing.

4.5. TestStrategyPacket

Produced by:
Behavior-derived test strategy stage

Used by:
- Test implementation stage
- Verification stage
- Judge stage
- Final report stage

Purpose:
Define tests from the BehaviorModel and PseudocodePacket before test files are written.

The TestStrategyPacket should not be a loose list of test ideas. It should map behavior to test responsibilities.

Required content:
- behaviors to protect
- state variables to verify
- events to test
- derived values to verify
- invariants to protect
- boundaries and partitions to cover
- external contracts to validate
- state transitions to test
- relevant failure modes
- test matrix
- test level assignment
- required verification commands
- coverage gaps
- risks not covered and why

Each proposed test should trace to at least one of:
- behavior
- invariant
- state transition
- derived value
- boundary
- external contract
- failure mode

Test levels:
- unit
- component
- integration
- end-to-end

Should not contain:
- full test file implementation unless the workflow explicitly combines strategy and implementation
- production code changes
- unrelated test expansion
- copied test cases that do not trace to behavior

Completion criteria:
The TestStrategyPacket is complete when a testing agent can implement tests without independently reinterpreting the feature.

4.9. JudgeReport

Produced by:
Judge stage

Used by:
- Final report stage
- workflow routing if the run needs correction

Purpose:
Review implementation, tests, and verification against the earlier artifacts.

The JudgeReport determines whether the run can be considered successful.

Required content:
- artifacts reviewed
- implementation vs PseudocodePacket assessment
- tests vs TestStrategyPacket assessment
- behavior coverage assessment
- architecture alignment assessment
- scope assessment
- verification evidence assessment
- risks and gaps
- verdict
- recommended next stage if not PASS

Possible verdicts:
- PASS
- DESIGN_INCOMPLETE
- PSEUDOCODE_INCOMPLETE
- IMPLEMENTATION_MISMATCH
- TEST_COVERAGE_INCOMPLETE
- ARCHITECTURE_MISMATCH
- NEED_CONTEXT
- SCOPE_VIOLATION
- NEED_VERIFICATION
- BLOCKED

Should not contain:
- new implementation patches
- unsupported pass verdict
- vague approval without checking artifacts
- hidden or minimized verification gaps

Completion criteria:
The JudgeReport is complete when it gives a clear verdict and, if needed, identifies the stage that should be revisited.

6. Artifact completeness rules

An artifact is complete when:
- it states its inputs
- it fulfills the purpose of its stage
- it is specific to the current request and project context
- it identifies uncertainty honestly
- it enables the next stage to proceed without redoing the previous stage

An artifact is incomplete when:
- it only restates the user request
- it skips required reasoning for the stage
- it contains unrelated content
- it makes unsupported claims
- it hides uncertainty
- it jumps ahead into later-stage work
- it is too vague to guide the next stage

The CLI does not need to judge completeness automatically in v0.1.0.
The judge prompt should evaluate completeness when necessary.

- PseudocodePacket

ImplementationReport depends on:
- ArchitectureContextPacket
- BehaviorModel
- PseudocodePacket

TestImplementationReport depends on:
- TestStrategyPacket
- BehaviorModel
- PseudocodePacket
- ImplementationReport when available

VerificationReport depends on:
- TestStrategyPacket
- ImplementationReport
- TestImplementationReport

JudgeReport depends on:
- all relevant prior artifacts

FinalReport depends on:
- JudgeReport
- VerificationReport
- ImplementationReport
- TestImplementationReport
- major design artifacts


8. Scope and authority of artifacts

RequestBrief authority:
Defines what the user asked for and what counts as success.

ArchitectureContextPacket authority:
Defines what project context was inspected and what architecture should be respected.

BehaviorModel authority:
Defines intended behavior.

PseudocodePacket authority:
Defines the implementation-neutral design contract.

TestStrategyPacket authority:
Defines what tests should protect.

ImplementationReport authority:
Defines what code actually changed.

TestImplementationReport authority:
Defines what tests actually changed.

VerificationReport authority:
Defines what was actually run.

JudgeReport authority:
Defines whether the run satisfied the design and verification requirements.

FinalReport authority:
Communicates the outcome to the user.


9. Updating artifacts

Artifacts may need to be updated if later stages reveal a contradiction.

Rules:
- Do not silently contradict an earlier artifact.
- If implementation reveals the PseudocodePacket is impossible or wrong, return to the pseudocode stage.
- If tests reveal the BehaviorModel is incomplete, return to the behavior model stage.
- If the project context was insufficient, return to the architecture context stage.
- If verification fails, update the relevant report and route to the appropriate prior stage.
- If scope changes, update RequestBrief or stop and ask for a new run depending on severity.

The workflow should preserve previous versions or note the correction in the updated artifact.

v0.1.0 can handle this manually through JudgeReport recommendations.


10. Artifact naming conventions

Use lowercase kebab-case filenames.

Examples:
request-brief.txt
architecture-context-packet.txt
behavior-model.txt
pseudocode-packet.txt
test-strategy-packet.txt
implementation-report.txt
test-implementation-report.txt

judge-report.txt
final-report.txt

Prompt files should also use lowercase kebab-case:

02-architecture-context.prompt.txt
03-behavior-model.prompt.txt
04-pseudocode-packet.prompt.txt
05-test-strategy.prompt.txt
06-implementation.prompt.txt
07-test-implementation.prompt.txt
08-verification.prompt.txt
09-judge.prompt.txt
10-final-report.prompt.txt

The artifact file and prompt file names should make the workflow order obvious.


11. Artifact use in generated prompts

A generated stage prompt should not include all previous artifacts in full unless required.

Preferred prompt input behavior:
- include the current stage name
- list required artifacts by filename
- include compact summaries if available
- instruct the coding agent to read the full artifacts from the run folder if needed
- include only prior artifacts required for the current stage

For example:
The pseudocode stage should use:
- request-brief.txt
- architecture-context-packet.txt
- behavior-model.txt

It should not need:
- implementation-report.txt
- verification-report.txt
- final-report.txt

The implementation stage should use:
- request-brief.txt
- architecture-context-packet.txt
- behavior-model.txt
- pseudocode-packet.txt

The test implementation stage should use:
- behavior-model.txt
- pseudocode-packet.txt
- test-strategy-packet.txt
- implementation-report.txt if code has already changed

This keeps prompts small and focused.


12. v0.1.0 artifact success criteria

The artifact system is successful if:

- each workflow stage produces a named text artifact
- the next stage can consume the artifact without redoing prior work
- implementation and test implementation share the same PseudocodePacket
- test strategy is derived from BehaviorModel and PseudocodePacket
- verification evidence is separated from claims
- judge can compare actual work against intended behavior
- final report can summarize the full run honestly
- the user can inspect and resume a run from files on disk

The artifact system should make the workflow traceable without making the CLI complicated.
