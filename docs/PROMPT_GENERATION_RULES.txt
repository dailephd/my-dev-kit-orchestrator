PROMPT_GENERATION_RULES.txt

Project name:
my-dev-kit-orchestrator

Document purpose:
This document defines how my-dev-kit-orchestrator generates prompts for coding agents.

The goal is to generate prompts that are focused, stage-specific, and easy for a coding agent to follow.

my-dev-kit-orchestrator should not generate one giant master prompt. The workflow can be large internally, but each prompt should only contain the instructions needed for the current stage.

The CLI owns the workflow structure.
The prompt owns only the current stage instruction.
The artifacts carry context between stages.


1. Core prompt generation principle

Generated prompts should be small, bounded, and stage-specific.

A prompt should tell the coding agent:

- what stage it is executing
- what input artifacts to use
- what task to perform
- what output artifact to produce
- what not to do in this stage
- where to save or return the result

A generated prompt should not try to teach the entire project philosophy every time.

The coding agent should not receive:
- the entire workflow architecture
- all workflow modes
- all artifact contracts
- all test design rules
- all possible failure categories
- unrelated scenario details
- instructions for stages that are not active

Prompt generation should preserve focus.


2. Why prompts must stay bounded

Large prompts can reduce coding-agent performance.

Problems caused by oversized prompts:

1. Attention dilution
   When many instructions are included, the coding agent may follow obvious surface rules while missing the main task.

2. Priority confusion
   Too many constraints can create unclear tradeoffs. The agent may not know whether to prioritize minimal change, completeness, architecture preservation, test coverage, or speed.

3. Anchoring
   If reusable prompt text contains concrete scenario details, the coding agent may imitate those details instead of applying the process to the current task.

4. Process over substance
   If a prompt is too procedural, the agent may produce a well-formatted report without making strong engineering decisions.

5. Premature stage mixing
   If a prompt includes instructions for design, implementation, testing, and verification at once, the agent may skip gates and blend stages.

6. Mechanical template filling
   If every prompt forces every category, the agent may fill irrelevant sections instead of deciding what matters for the current task.

The prompt system should avoid these failure modes by keeping each prompt focused on one stage.


3. Prompt generation model

my-dev-kit-orchestrator should generate one prompt per stage.

The standard prompt shape is:

Stage:
<stage name>

Workflow mode:
<feature | repair | test | refactor | harden>

Inputs:
<required artifacts and project context>

Task:
<what the coding agent must do now>

Output:
<required artifact name and expected file>

Stop conditions:
<what the coding agent must not do in this stage>

Return format:
<how the coding agent should return or save the output>

This structure should be consistent across stages.


4. What the CLI should inject into prompts

The CLI should inject:

- current stage name
- workflow mode
- run ID
- project root
- original request or compact RequestBrief summary
- required input artifact filenames
- compact summaries of required prior artifacts when available
- current stage task
- required output artifact name
- output file path
- stop conditions
- stage gate reminders

The CLI should not inject:
- unrelated workflow modes
- full project documentation
- every artifact contract
- reusable concrete scenario details
- every possible test category
- instructions for future stages
- implementation instructions before implementation stage
- test implementation instructions before test implementation stage
- verification claims before verification stage


5. Prompt length policy

Generated prompts should be concise enough to keep the agent focused.

v0.1.0 target:
- most stage prompts should fit on one screen when possible
- include only the current stage and required inputs
- refer to full artifacts by filename instead of pasting everything repeatedly
- include compact artifact summaries when helpful
- avoid long repeated rule lists

A prompt may be longer when:
- the stage is inherently complex
- the required artifact has a detailed structure
- the prior artifacts are short and must be included for continuity
- the coding agent has no filesystem access and must receive artifact content inline

Even when a prompt is longer, it should remain stage-specific.


6. Stage gates in prompts

Prompts should enforce workflow order through stop conditions.

A stop condition prevents the coding agent from doing work that belongs to a later stage.

Required gates:

Gate 1:
The behavior model stage must not write pseudocode, production code, or test files.

Gate 2:
The pseudocode packet stage must not write production code or test files.

Gate 3:
The test strategy stage must not write test files unless the workflow explicitly combines strategy and implementation for a small task.

Gate 4:
The implementation stage must not redesign the feature unless it reports that the pseudocode packet is impossible or inconsistent with the architecture context.

Gate 5:
The test implementation stage must follow the TestStrategyPacket and should not invent a separate test strategy.

Gate 6:
The verification stage must not claim success without command evidence.

Gate 7:
The judge stage must not approve a run unless implementation, tests, and verification are compared against the design artifacts.

Gate 8:
If required context is missing, the agent should report the missing context instead of guessing.

These gates should appear in the relevant stage prompts, not as one giant global list in every prompt.


7. Strictness policy

The prompt system should be strict only where strictness improves correctness.

Strict areas:
- stage boundaries
- required input artifacts
- required output artifact
- file modification scope
- verification evidence
- explicit reporting of blockers
- explicit reporting of uncertainty
- traceability between design, pseudocode, tests, and verification

Flexible areas:
- how the agent reasons within a stage
- which state variables are relevant
- which failure modes are relevant
- which test levels are appropriate
- whether existing architecture supports the design
- whether a boundary needs a test
- whether a risk is worth covering now or should be reported as uncovered

The prompt should not force irrelevant sections. It should require the agent to decide what matters for the current task and justify important omissions.


8. No reusable concrete scenario templates

Generic prompts should not contain concrete reusable scenario details.

The current request and current project context should supply the specifics.

A stage prompt should say:

- derive behavior from the RequestBrief and ArchitectureContextPacket
- derive pseudocode from the BehaviorModel
- derive tests from the BehaviorModel and PseudocodePacket
- inspect the current architecture before proposing changes
- report missing context when needed

A stage prompt should not contain unrelated concrete feature details that could steer the coding agent away from the current task.

This rule is especially important for test strategy prompts. The test strategy prompt should describe the reasoning method generically and require the agent to apply that method to the current feature.


9. Artifact references instead of repeated context

Prompts should use artifacts as the source of continuity.

A prompt should normally reference prior artifacts by filename.

For example, the pseudocode packet prompt should use:
- artifacts/request-brief.txt
- artifacts/architecture-context-packet.txt
- artifacts/behavior-model.txt

The implementation prompt should use:
- artifacts/request-brief.txt
- artifacts/architecture-context-packet.txt
- artifacts/behavior-model.txt
- artifacts/pseudocode-packet.txt

The test implementation prompt should use:
- artifacts/behavior-model.txt
- artifacts/pseudocode-packet.txt
- artifacts/test-strategy-packet.txt
- artifacts/implementation-report.txt if implementation already happened

The prompt should include compact summaries when useful, but it should not paste every prior artifact in full unless needed.


10. Prompt output requirements

Every generated prompt should require a named output artifact.

The prompt should specify:

- artifact name
- output filename
- required sections
- whether the artifact may be incomplete
- how to report blockers
- how to report uncertainty

A stage prompt should not ask for vague output such as:
- “analyze this”
- “fix this”
- “write tests”
- “improve the code”

It should ask for a specific artifact such as:
- RequestBrief
- ArchitectureContextPacket
- BehaviorModel
- PseudocodePacket
- TestStrategyPacket
- ImplementationReport
- TestImplementationReport
- VerificationReport
- JudgeReport
- FinalReport


11. Prompt stop conditions

Every generated prompt should include stop conditions.

Stop conditions define what the agent must not do in the current stage.

Common stop conditions:

Request brief stage:
- do not inspect code deeply
- do not write pseudocode
- do not write tests
- do not implement code

Architecture context stage:
- do not redesign the feature
- do not write pseudocode
- do not write tests
- do not implement code

Behavior model stage:
- do not write production code
- do not write test files
- do not convert the behavior model into implementation-specific code

Pseudocode packet stage:
- do not write production code
- do not write test files
- do not modify files

Test strategy stage:
- do not write production code
- do not write test files unless explicitly allowed
- do not propose tests that cannot be traced to behavior, invariant, transition, derived value, boundary, contract, or failure mode

Implementation stage:
- do not modify files outside justified scope
- do not create parallel architecture
- do not claim verification success
- do not ignore contradictions in the PseudocodePacket or ArchitectureContextPacket

Test implementation stage:
- do not replace the TestStrategyPacket with a new unrelated strategy
- do not change production behavior unless reporting a blocker
- do not claim tests passed unless run in verification stage or command evidence is included

Verification stage:
- do not claim success without command output
- do not hide failed commands
- do not skip required checks without explanation

Judge stage:
- do not approve without comparing artifacts
- do not rewrite code
- do not hide verification gaps

Final report stage:
- do not exaggerate success
- do not omit unresolved risks
- do not claim unrun checks passed


12. Stage prompt templates

12.1. Request brief prompt shape

Stage:
Request brief

Use:
- original request
- workflow mode
- project root if known

Task:
Produce artifacts/request-brief.txt.

The RequestBrief must define:
- original request
- requested change
- target area if known
- user-visible or external behavior
- constraints
- non-goals
- success criteria
- ambiguity or missing information
- expected next stage

Stop conditions:
- do not inspect code deeply
- do not write pseudocode
- do not write tests
- do not implement code


12.2. Architecture context prompt shape

Stage:
Architecture context

Use:
- artifacts/request-brief.txt
- project root
- my-dev-kit if available

Task:
Produce artifacts/architecture-context-packet.txt.

The ArchitectureContextPacket must identify:
- relevant files
- relevant symbols
- relevant components, modules, routes, commands, services, or boundaries
- relevant tests
- relevant docs
- state owners
- data owners
- upstream dependencies
- downstream consumers
- existing patterns to preserve
- likely files or modules involved
- context gaps or uncertainty
- selection rationale

When my-dev-kit is available:
- use graph-guided retrieval before broad file reading
- retrieve focused context around relevant files and symbols
- inspect tests and docs related to the current request

Stop conditions:
- do not redesign the feature
- do not write pseudocode
- do not write tests
- do not implement code


12.3. Behavior model prompt shape

Stage:
Behavior model

Use:
- artifacts/request-brief.txt
- artifacts/architecture-context-packet.txt

Task:
Produce artifacts/behavior-model.txt.

The BehaviorModel must define:
- behavior summary
- externally visible behavior
- internal supporting behavior
- state variables
- inputs
- events
- derived values
- invariants
- valid states
- invalid states
- boundaries and partitions
- empty states
- error states
- loading or pending states when relevant
- external contracts
- state transitions
- behavior to preserve
- behavior intentionally changed
- unresolved design questions

Stop conditions:
- do not write production code
- do not write test files
- do not produce implementation-specific code unless required to explain architecture constraints


12.4. Pseudocode packet prompt shape

Stage:
Pseudocode packet

Use:
- artifacts/request-brief.txt
- artifacts/architecture-context-packet.txt
- artifacts/behavior-model.txt

Task:
Produce artifacts/pseudocode-packet.txt.

The PseudocodePacket must convert the BehaviorModel into implementation-neutral design.

It must define:
- behavior references
- data flow
- state flow
- state transitions
- derived-value rules
- validation rules
- empty-state handling
- error-state handling
- external contract handling
- component, module, service, route, command, or helper contracts
- acceptance criteria
- likely files or modules to modify
- implementation constraints
- assumptions preserved
- assumptions removed or guarded
- unresolved implementation questions

Stop conditions:
- do not write production code
- do not write test files
- do not modify files


12.5. Test strategy prompt shape

Stage:
Behavior-derived test strategy

Use:
- artifacts/request-brief.txt
- artifacts/architecture-context-packet.txt
- artifacts/behavior-model.txt
- artifacts/pseudocode-packet.txt

Task:
Produce artifacts/test-strategy-packet.txt.

The TestStrategyPacket must derive test responsibilities from the BehaviorModel and PseudocodePacket.

It must identify:
- behavior under test
- participating layers
- state variables
- events
- derived values
- invariants
- boundaries and partitions
- external contracts
- state transitions when relevant
- relevant failure modes
- existing tests
- test matrix
- test level assignment
- required verification commands
- coverage gaps
- risks not covered and why

Each test responsibility must trace to at least one of:
- behavior
- invariant
- state transition
- derived value
- boundary
- external contract
- failure mode

Stop conditions:
- do not write production code
- do not write test files in this stage unless explicitly allowed
- do not invent behavior not supported by prior artifacts
- do not include test responsibilities that cannot be traced to the behavior model or pseudocode packet


12.6. Implementation prompt shape

Stage:
Implementation

Use:
- artifacts/request-brief.txt
- artifacts/architecture-context-packet.txt
- artifacts/behavior-model.txt
- artifacts/pseudocode-packet.txt

Task:
Implement the PseudocodePacket in the current project and produce artifacts/implementation-report.txt.

The ImplementationReport must include:
- files read
- files changed
- behavior implemented
- pseudocode sections implemented
- assumptions preserved
- assumptions removed or guarded
- deviations from the PseudocodePacket
- reason for each deviation
- blockers encountered
- unresolved risks
- tests that should be run
- notes for test implementation stage

Stop conditions:
- do not broaden scope without reporting a blocker
- do not create parallel architecture
- do not claim verification success without command evidence
- do not ignore contradictions in prior artifacts


12.7. Test implementation prompt shape

Stage:
Test implementation

Use:
- artifacts/behavior-model.txt
- artifacts/pseudocode-packet.txt
- artifacts/test-strategy-packet.txt
- artifacts/implementation-report.txt if available

Task:
Implement tests from the TestStrategyPacket and produce artifacts/test-implementation-report.txt.

The TestImplementationReport must include:
- test files changed
- tests added
- tests updated
- behaviors covered
- invariants covered
- state transitions covered
- derived values covered
- boundaries covered
- external contracts covered
- failure modes covered
- TestStrategyPacket items not implemented
- reason for each missing test
- verification commands to run

Stop conditions:
- do not replace the TestStrategyPacket with a new unrelated strategy
- do not change production behavior unless reporting a blocker
- do not claim tests passed unless command evidence is included


12.8. Verification prompt shape

Stage:
Verification

Use:
- artifacts/test-strategy-packet.txt
- artifacts/implementation-report.txt
- artifacts/test-implementation-report.txt

Task:
Run the required verification commands and produce artifacts/verification-report.txt.

The VerificationReport must include:
- commands run
- working directory for each command
- exit codes
- pass/fail status
- output summary
- failed tests if any
- skipped checks
- reason for skipped checks
- environment notes if relevant
- remaining verification gaps

Stop conditions:
- do not claim checks passed unless command evidence exists
- do not hide failed commands
- do not omit skipped required checks


12.9. Judge prompt shape

Stage:
Judge

Use:
- all relevant prior artifacts

Task:
Produce artifacts/judge-report.txt.

The JudgeReport must assess:
- implementation vs PseudocodePacket
- tests vs TestStrategyPacket
- behavior coverage
- architecture alignment
- scope control
- verification evidence
- risks and gaps

Verdict must be one of:
- PASS
- DESIGN_INCOMPLETE
- PSEUDOCODE_INCOMPLETE
- IMPLEMENTATION_MISMATCH
- TEST_COVERAGE_INCOMPLETE
- ARCHITECTURE_MISMATCH
- NEED_CONTEXT
- SCOPE_VIOLATION
- NEED_VERIFICATION
- BLOCKED

Stop conditions:
- do not rewrite code
- do not approve without verification evidence
- do not hide uncertainty or gaps


12.10. Final report prompt shape

Stage:
Final report

Use:
- artifacts/judge-report.txt
- artifacts/verification-report.txt
- major design and implementation artifacts

Task:
Produce artifacts/final-report.txt.

The FinalReport must include:
- original request
- workflow mode
- run ID
- stages completed
- behavior designed
- architecture context used
- pseudocode summary
- tests designed
- implementation summary
- tests added or updated
- verification summary
- judge verdict
- unresolved risks
- follow-up recommendations if needed

Stop conditions:
- do not exaggerate success
- do not omit failed or skipped verification
- do not hide unresolved risks


13. Prompt generation by workflow mode

The CLI should select a stage sequence based on mode.

Feature mode:
- request brief
- architecture context
- behavior model
- pseudocode packet
- test strategy
- implementation
- test implementation
- verification
- judge
- final report

Repair mode:
- observed behavior report
- architecture context
- behavior trace
- divergence report
- correction design
- regression test strategy
- implementation
- test implementation
- verification
- judge
- final report

Test mode:
- test target brief
- architecture context
- behavior reconstruction or behavior model
- pseudocode summary or pseudocode packet
- test strategy
- test implementation
- verification
- judge
- final report

Refactor mode:
- refactor brief
- architecture context
- existing behavior map
- preserved invariant list
- compatibility test strategy
- refactor pseudocode packet
- implementation
- test implementation
- verification
- judge
- final report

Harden mode:
- hardening brief
- architecture context
- assumption report
- failure mode matrix
- guard pseudocode packet
- resilience test strategy
- implementation
- test implementation
- verification
- judge
- final report

Each mode should still use the same prompt generation principles:
- one stage at a time
- named artifact output
- stop conditions
- compact context
- no unrelated mode instructions


14. How prompts should handle missing artifacts

If a prompt requires an artifact that is missing, the generated prompt should not pretend the artifact exists.

The CLI should either:
- print the prompt for the missing earlier stage
- or print a blocker message explaining which artifact is missing

A stage prompt may include:

Required input missing:
<artifact filename>

Action:
Return to the stage that produces this artifact before continuing.

The coding agent should not be asked to proceed without required context unless the workflow explicitly supports reconstruction.


15. How prompts should handle uncertainty

Generated prompts should require the coding agent to report uncertainty.

Uncertainty should be explicit, not hidden.

A stage artifact may include:
- context gaps
- unresolved design questions
- implementation blockers
- verification gaps
- untested risks

The prompt should not force the agent to invent answers where evidence is missing.

Useful instruction:
If required information is unavailable after reasonable inspection, report the missing information and explain how it affects the next stage.


16. How prompts should handle scope

Implementation and test implementation prompts must include scope boundaries.

Scope instructions should say:
- modify only files justified by the PseudocodePacket, TestStrategyPacket, or ArchitectureContextPacket
- do not perform unrelated cleanup
- do not introduce parallel architecture
- do not broaden the feature without reporting the reason
- if broader change is required, report a blocker or route back to the relevant upstream stage

Scope control should be strict during implementation and test implementation.
Scope should be more flexible during architecture context and behavior modeling, where discovery is expected.


17. How prompts should handle my-dev-kit

The architecture context prompt should invoke my-dev-kit when available.

The prompt should instruct the coding agent to:
- use project-local my-dev-kit if specified
- use npm package execution if that is the configured access path
- index or refresh project context if necessary
- perform focused retrieval before broad reading
- retrieve graph-guided context around relevant symbols and files
- identify relevant tests and docs
- summarize the selection rationale

The prompt should not require my-dev-kit automation in v0.1.0.
It should instruct the coding agent how to use it when available.

Later versions may automate this stage directly.

19. Prompt quality checklist

A generated prompt is good if:
- it names exactly one active stage
- it lists required input artifacts
- it defines exactly one required output artifact
- it gives enough instruction to complete the stage
- it includes stop conditions
- it avoids unrelated workflow modes
- it avoids unnecessary concrete scenario details
- it does not mix implementation with design stages
- it does not mix verification claims with implementation stages
- it is short enough to keep the coding agent focused
- it tells the agent what to do if context is missing

A generated prompt is poor if:
- it tries to run the whole workflow at once
- it includes every project rule
- it includes unrelated mode instructions
- it includes scenario details not from the current request
- it asks for code before pseudocode exists
- it asks for tests before test strategy exists
- it has no stop conditions
- it has no named output artifact
- it encourages broad guessing
- it rewards formatting over engineering substance


20. v0.1.0 prompt generation success criteria

The prompt generation system is successful if:

- the user can start a workflow with one command
- the user can print the next stage prompt with one command
- each prompt is bounded to one stage
- each prompt produces one named artifact
- the coding agent is not overloaded with the full workflow
- the workflow gates prevent premature implementation
- the PseudocodePacket becomes the shared design source for code and tests
- the TestStrategyPacket is produced before test implementation
- verification evidence is separated from implementation claims
- the judge can compare actual work against intended design
- the user can inspect the run folder and understand the workflow state

The prompt generator should make disciplined design-first work easier than writing one large unstructured coding-agent prompt.
