Project name:
my-dev-kit-orchestrator

Purpose:
my-dev-kit-orchestrator is a CLI-first workflow tool for design-first software development with coding agents.

Its purpose is to help a coding agent write correct code the first time by forcing a structured sequence before implementation:

request
→ project context
→ behavior model
→ pseudocode contract
→ behavior-derived test strategy
→ implementation
→ test implementation
→ verification
→ review

The project is not centered on repairing broken code after the fact. Repair is one supported mode, but the primary purpose is first-pass correctness: converting a feature request or software-change request into a shared design artifact that both the coding agent and the testing agent use.

The central idea:
The coding agent and testing agent should not independently infer what the feature means.

They should both work from the same behavior model and pseudocode contract.

That contract becomes the source of truth for:
- what should be built
- what should not be built
- what behavior must be preserved
- what state and data flows matter
- what tests must prove
- what verification evidence is required


Why this project exists:
Coding agents often fail because they start writing code too early. They may:
- modify code before understanding the current architecture
- implement from vague requirements
- create a parallel architecture instead of extending the existing one
- write tests after implementation as an afterthought
- test only surface interactions
- miss state transitions, boundaries, contracts, and derived values
- make broad changes outside the intended scope
- claim correctness without verification evidence

my-dev-kit-orchestrator exists to reduce those failures by controlling the order of work and by producing stage-specific prompts and handoff artifacts.


Relationship to my-dev-kit:
my-dev-kit remains the code understanding and retrieval substrate.

my-dev-kit is responsible for:
- indexing a project
- finding relevant files and symbols
- retrieving graph-guided context
- exposing code structure
- helping the agent understand the existing architecture

my-dev-kit-orchestrator is responsible for:
- selecting the appropriate development workflow
- generating bounded prompts for each stage
- defining handoff artifacts
- ensuring code and tests come from the same design context
- preserving stage order
- producing final verification and review prompts

The relationship is:

my-dev-kit
= code retrieval and project understanding

my-dev-kit-orchestrator
= design-to-code workflow orchestration

my-dev-kit-lab
= security validation, adversarial testing, and package-safety work for my-dev-kit

my-dev-kit-orchestrator should be a separate project from my-dev-kit-lab because it is not primarily a security-validation framework. It is a general software-development workflow tool for coding agents.


Core philosophy:
The workflow should be design-first, not code-first.

The tool should not ask a coding agent to simply “implement this” from a raw request. It should first generate a sequence of artifacts:

1. RequestBrief
   A compact statement of the requested software change, including goal, constraints, non-goals, and success criteria.

2. ArchitectureContextPacket
   A focused summary of the relevant project context, retrieved using my-dev-kit and graph-guided retrieval when available.

3. BehaviorModel
   A model of the intended behavior, including externally visible behavior, state variables, events, derived values, invariants, boundaries, contracts, and expected state transitions.

4. PseudocodePacket
   An implementation-neutral pseudocode contract that converts the behavior model into precise data flow, state flow, validation rules, component/module contracts, and acceptance criteria.

5. TestStrategyPacket
   A behavior-derived test plan that maps behavior, invariants, transitions, boundaries, and external contracts to concrete test responsibilities at the narrowest useful test level.

6. ImplementationReport
   A report from the coding stage describing what was changed, how the pseudocode was implemented, and whether there were deviations.

7. TestImplementationReport
   A report from the testing stage describing what tests were added or changed and what behavior each test protects.

8. VerificationReport
   Evidence from executed commands, including test, typecheck, lint, build, or smoke-test results as appropriate.

9. JudgeReport
   A review artifact that compares the implementation and tests against the behavior model, pseudocode packet, and test strategy.

10. FinalReport
   A user-facing completion report summarizing what was done, what was verified, and what risks remain.


Important design principle:
The workflow should be large internally but small at the prompt level.

my-dev-kit-orchestrator should not generate one giant prompt containing every workflow rule. Large prompts can dilute attention, overconstrain the coding agent, and cause the agent to follow process mechanically instead of solving the current task.

Instead, the CLI should behave like a workflow state machine:

- The CLI owns the workflow sequence.
- Each stage prompt is short and bounded.
- Each stage prompt uses only the artifacts needed for that stage.
- Each stage produces a named artifact.
- The next stage consumes only the relevant prior artifacts.
- Strictness is used at stage gates, output requirements, and scope boundaries.
- Reasoning inside each stage remains flexible enough to adapt to the current project.

A generated prompt should generally contain:
1. stage name
2. available inputs
3. task for this stage
4. required output artifact
5. stop conditions

The tool should avoid reusable concrete scenario templates inside generic prompts. The current task and current project context should determine the specific details.


Supported workflow modes:
The project should support a small number of explicit modes.

1. feature
   For building a new feature or extending an existing feature.

   Flow:
   request
   → architecture context
   → behavior model
   → pseudocode packet
   → behavior-derived test strategy
   → implementation
   → test implementation
   → verification
   → judge

2. repair
   For observed behavior that does not match the intended design.

   This mode is design-conformance repair, not raw bug hunting.

   Flow:
   observed behavior
   → behavior/design lookup
   → pseudocode/runtime divergence analysis
   → correction design
   → regression-focused test strategy
   → implementation
   → verification
   → judge

   Repair should begin by connecting the observed behavior to the behavior model or pseudocode packet when those artifacts exist. The goal is to find the first divergence between intended behavior and actual runtime behavior.

3. test
   For designing or implementing comprehensive tests for an existing feature or workflow.

   Flow:
   test target
   → architecture context
   → behavior reconstruction
   → pseudocode lookup or creation
   → behavior-derived test strategy
   → test implementation
   → verification
   → judge

4. refactor
   For changing code structure while preserving behavior.

   Flow:
   refactor request
   → architecture context
   → existing behavior map
   → preserved invariants
   → compatibility test strategy
   → refactor pseudocode
   → implementation
   → verification
   → judge

5. harden
   For improving robustness, validation, and failure handling.

   Flow:
   hardening request
   → architecture context
   → assumption analysis
   → failure-mode analysis
   → guard/error-state pseudocode
   → resilience test strategy
   → implementation
   → verification
   → judge


CLI direction:
The CLI should be simple and explicit.

The project command should be readable and memorable, not abbreviated.

Preferred command name:
my-dev-kit-orchestrator

The command surface should stay small:

my-dev-kit-orchestrator init
my-dev-kit-orchestrator start "<request>"
my-dev-kit-orchestrator start --mode feature "<request>"
my-dev-kit-orchestrator start --mode repair "<observed behavior>"
my-dev-kit-orchestrator start --mode test "<test target>"
my-dev-kit-orchestrator start --mode refactor "<refactor goal>"
my-dev-kit-orchestrator start --mode harden "<hardening goal>"
my-dev-kit-orchestrator status
my-dev-kit-orchestrator prompt
my-dev-kit-orchestrator prompt <stage>
my-dev-kit-orchestrator list

The CLI should not expose every internal stage as a top-level command. The user should not need to manually operate many separate agents. The user should start a workflow, inspect status, and print the next prompt.


Default run directory:
The tool should create workflow runs under:

.my-dev-kit-orchestrator/runs/

Each run should contain:
- the original request
- generated stage prompts
- generated or pasted artifacts
- run metadata
- final reports

The exact file layout can be refined later, but the project should preserve the principle that each run is inspectable and resumable.


Prompt generation principle:
The CLI should generate stage-specific prompts, not a single master prompt.

Each stage prompt should:
- state the current stage
- list the required input artifacts
- define the task for that stage
- define the required output artifact
- define what the agent must not do in that stage
- remain generic and task-adaptive
- avoid injecting unrelated workflow modes
- avoid reusable concrete scenario templates
- avoid excessive rule lists

The workflow should rely on artifacts and stage gates rather than overwhelming the coding agent with long instructions.


Behavior-derived test design:
The test strategy stage must derive tests from the behavior model.

The process is:

behavior model
→ state variables
→ events
→ derived values
→ invariants
→ boundaries and partitions
→ external contracts
→ state-transition matrix
→ test matrix
→ test level assignment

The test strategy should not begin from a loose list of individual test cases. It should first model the behavior and then derive tests from that model.

Each proposed test should trace back to at least one of:
- behavior
- invariant
- state transition
- derived value
- boundary
- external contract
- failure mode

Tests should be assigned to the narrowest useful level:
- unit
- component
- integration
- end-to-end

The test strategy should also state which risks are not covered and why.


Design map concept:
Over time, the project should support a design map that links:

feature intent
→ behavior IDs
→ pseudocode steps
→ implementation symbols
→ tests
→ runtime observations

This map allows the same system to support both first-pass feature implementation and later design-conformance repair.

For feature work, the workflow moves forward:

feature intent
→ behavior model
→ pseudocode
→ code
→ tests
→ verification

For repair work, the workflow moves backward from observation:

observed behavior
→ runtime trace
→ code path
→ pseudocode step
→ behavior model
→ corrected design/code/test artifact

This allows the system to locate the first divergence between intended behavior and observed behavior instead of relying on unfocused code searching.


What this project is not:
my-dev-kit-orchestrator is not:
- a debugging-only tool
- a giant prompt template
- an example-driven prompt system
- a replacement for my-dev-kit
- a replacement for the coding agent
- a fully automated multi-agent platform in the first version
- a security validation framework
- a tool that should expose many low-level commands to the user

The first version should be a practical CLI prompt-orchestration tool.

It should help the user:
- start a workflow
- generate the correct next prompt
- preserve artifacts
- maintain workflow order
- use my-dev-kit context retrieval when needed
- guide the coding agent through design, pseudocode, test strategy, implementation, and verification


v0.1.0 target:
The first version should prioritize clarity and usefulness over automation.

v0.1.0 should include:
- explicit CLI command name
- simple command surface
- local workflow run folders
- feature, repair, test, refactor, and harden modes
- stage-specific prompt generation
- artifact handoff structure
- graph-guided retrieval stage support
- behavior model stage
- pseudocode packet stage
- behavior-derived test strategy stage
- implementation prompt stage
- test implementation prompt stage
- verification and judge prompt stage
- final report template

v0.1.0 does not need to run the LLM directly.
v0.1.0 does not need full JSON schema validation.
v0.1.0 does not need full automation of my-dev-kit command execution.
v0.1.0 does not need complex agent routing.

The most important v0.1.0 outcome:
The user can run:

my-dev-kit-orchestrator start "<software change request>"
my-dev-kit-orchestrator prompt

Then paste the generated prompt into a coding agent and proceed stage by stage through a design-first workflow.


Long-term direction:
Later versions may add:
- artifact validation
- design map generation
- behavior ID tracking
- code/test trace links
- deeper my-dev-kit integration
- automatic context retrieval
- run resumption
- structured reports
- optional LLM provider integration
- CI-friendly verification summaries

These should come after the core workflow and prompt-generation design is stable.


Core success criterion:
my-dev-kit-orchestrator succeeds if it helps a coding agent avoid premature implementation.

A successful run should show that:
- the requested behavior was understood before code was written
- the current architecture was inspected before design decisions were made
- pseudocode was created before implementation
- tests were designed from behavior before test files were written
- code and tests shared the same design source
- verification evidence was collected
- remaining risks were stated clearly
