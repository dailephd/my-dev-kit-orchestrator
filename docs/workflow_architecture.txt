workflow_architecture.txt

Project name:
my-dev-kit-orchestrator

Document purpose:
This document defines the workflow architecture for my-dev-kit-orchestrator.

my-dev-kit-orchestrator is a CLI-first workflow tool that helps a user guide coding agents through design-first software development.

The workflow architecture is based on a simple principle:

The CLI owns the workflow.
The coding agent executes one bounded stage at a time.
Artifacts carry context between stages.
Implementation does not begin until behavior and pseudocode are defined.
Tests are derived from behavior before test files are written.


1. Core architecture

my-dev-kit-orchestrator should behave like a workflow state machine.

It should not generate one enormous prompt that contains every rule, every mode, and every reasoning framework.

Instead, the tool should:
- create a run folder
- choose a workflow mode
- generate stage-specific prompts
- track which artifacts exist
- identify the next stage
- print the next prompt when requested
- preserve all stage outputs in an inspectable run folder

The coding agent should receive only the prompt for the current stage.

The current stage prompt should include:
- stage name
- required inputs
- task for this stage
- required output artifact
- stop conditions

The prompt should not include unrelated workflow modes.
The prompt should not include the full project philosophy unless needed.
The prompt should not include reusable concrete scenarios.
The prompt should not ask the coding agent to complete multiple unrelated stages at once.


2. Workflow state machine

Each run has a mode and a current stage.

A run moves forward when the expected artifact for the current stage is produced.

Generic state shape:

Run
- run_id
- project_root
- mode
- request
- current_stage
- completed_stages
- pending_stages
- artifacts
- prompts
- reports
- status

Possible run statuses:
- created
- in_progress
- waiting_for_artifact
- blocked
- completed
- failed
- abandoned

Possible stage statuses:
- pending
- prompt_generated
- artifact_missing
- artifact_present
- blocked
- skipped
- completed

The CLI does not need full schema validation in v0.1.0, but it should still track expected artifact names and whether the corresponding artifact files exist.


3. Core workflow stages

The standard design-to-code sequence is:

1. Request brief
2. Architecture context
3. Behavior model
4. Pseudocode packet
5. Behavior-derived test strategy
6. Implementation
7. Test implementation
8. Verification
9. Judge
10. Final report

The core sequence is used directly by feature mode and adapted by other modes.

The high-level flow is:

request
→ architecture context
→ behavior model
→ pseudocode packet
→ test strategy
→ implementation
→ test implementation
→ verification
→ judge
→ final report

The important design rule:
The implementation and test implementation stages both depend on the same behavior model and pseudocode packet.

This prevents the coding agent and testing agent from independently interpreting the request.


4. Stage responsibilities

4.1. Request brief stage

Artifact produced:
RequestBrief

Purpose:
Convert the user request into a compact software-change brief.

The RequestBrief should define:
- requested change
- target project or repo
- target workflow, component, module, or command if known
- user-visible behavior or external behavior
- constraints
- non-goals
- success criteria
- ambiguity or missing information

This stage should not inspect code deeply.
This stage should not produce pseudocode.
This stage should not write tests.
This stage should not modify code.

The RequestBrief is the first binding artifact for the run.


4.2. Architecture context stage

Artifact produced:
ArchitectureContextPacket

Purpose:
Ground the workflow in the current project before design decisions are made.

This stage should use my-dev-kit when available.

The ArchitectureContextPacket should identify:
- relevant files
- relevant symbols
- relevant components or modules
- relevant routes, commands, services, or data boundaries
- relevant tests
- relevant docs
- state owners
- data owners
- upstream dependencies
- downstream consumers
- existing project patterns to preserve
- likely files or modules involved
- context gaps or uncertainties

This stage should use graph-guided retrieval when available.

This stage should avoid broad whole-file reading unless targeted retrieval is insufficient.

This stage should not redesign the feature.
This stage should not produce implementation pseudocode.
This stage should not write tests.
This stage should not modify code.

The ArchitectureContextPacket is a context artifact, not an implementation plan.


4.3. Behavior model stage

Artifact produced:
BehaviorModel

Purpose:
Define what the software should do before deciding how to implement it.

The BehaviorModel should describe:
- externally visible behavior
- internal behavior required to support it
- state variables
- external inputs
- internal inputs
- events
- derived values
- invariants
- boundaries
- valid states
- invalid states
- empty states
- error states
- loading or pending states when relevant
- external contracts
- state transitions
- behavior that must be preserved
- behavior that is intentionally changed

This stage should not write production code.
This stage should not write test files.
This stage should not make implementation-specific changes.

The BehaviorModel is the source artifact for both pseudocode and test strategy.


4.4. Pseudocode packet stage

Artifact produced:
PseudocodePacket

Purpose:
Convert the BehaviorModel into implementation-neutral pseudocode.

The PseudocodePacket should define:
- behavior IDs or behavior references when useful
- data flow
- state flow
- state transitions
- derived-value rules
- validation rules
- empty-state handling
- error-state handling
- external contract handling
- component, module, service, or command contracts
- helper function contracts when relevant
- acceptance criteria
- likely files or modules to modify
- constraints on implementation
- assumptions that must remain true
- assumptions that must be removed or guarded

The PseudocodePacket is the shared design contract for:
- implementation
- test implementation
- verification
- review

This stage should not write production code.
This stage should not write test files.
This stage should not modify files.

The PseudocodePacket should not be tied to unnecessary implementation details.
It should be precise enough to guide code and tests while remaining grounded in the existing architecture.


4.5. Behavior-derived test strategy stage

Artifact produced:
TestStrategyPacket

Purpose:
Derive tests from the BehaviorModel and PseudocodePacket.

The TestStrategyPacket should identify:
- behaviors to protect
- state variables that control behavior
- events that change state
- derived values to verify
- invariants to protect
- boundaries and partitions to cover
- external contracts to validate
- state transitions to test
- relevant failure modes
- appropriate test levels
- required verification commands
- uncovered risks

Each proposed test must trace to at least one of:
- behavior
- invariant
- state transition
- derived value
- boundary
- external contract
- failure mode

The test strategy should assign tests to the narrowest useful level:
- unit
- component
- integration
- end-to-end

This stage should not write test files unless the workflow explicitly combines test design and test implementation for a small task.

This stage should not implement feature code.

The output should be a strategy artifact, not a loose list of test ideas.


4.6. Implementation stage

Artifact produced:
ImplementationReport

Purpose:
Implement the PseudocodePacket using the ArchitectureContextPacket.

The implementation prompt should instruct the coding agent to:
- read the required files
- follow the existing architecture
- implement the PseudocodePacket
- preserve the BehaviorModel
- modify only justified files
- avoid creating parallel architecture
- avoid unrelated cleanup
- avoid scope expansion
- report blockers instead of guessing
- document deviations from the PseudocodePacket

The ImplementationReport should include:
- files changed
- behavior implemented
- pseudocode sections implemented
- assumptions added, removed, or guarded
- deviations from the PseudocodePacket
- blockers
- risks introduced
- tests that should be run

This stage may modify production code.
This stage should not replace the test strategy.
This stage should not claim success without verification.


4.7. Test implementation stage

Artifact produced:
TestImplementationReport

Purpose:
Write or update tests from the TestStrategyPacket.

The test implementation prompt should instruct the coding agent to:
- implement tests mapped to the TestStrategyPacket
- preserve traceability from tests to behavior
- prefer the narrowest useful test level
- cover required boundaries and transitions
- cover relevant external contracts
- report any strategy item that cannot be tested
- avoid shallow tests that do not protect behavior

The TestImplementationReport should include:
- tests added
- tests updated
- behavior covered
- invariants covered
- transitions covered
- contracts covered
- tests not implemented
- reason for missing tests
- commands required for verification

This stage may modify test files.
This stage should not change feature behavior unless a blocker is found and reported.


4.8. Verification stage

Artifact produced:
VerificationReport

Purpose:
Run the required commands and collect evidence.

The verification prompt should instruct the coding agent to run appropriate commands from prior artifacts and project conventions.

Possible command categories:
- targeted unit tests
- targeted component tests
- targeted integration tests
- relevant end-to-end tests
- typecheck
- lint
- build
- smoke checks

The VerificationReport should include:
- commands run
- exit codes
- pass/fail status
- relevant output summaries
- failed tests
- skipped checks
- reason for skipped checks
- remaining verification gaps

This stage should not claim checks passed unless command evidence exists.


4.9. Judge stage

Artifact produced:
JudgeReport

Purpose:
Review implementation and tests against the workflow artifacts.

The judge should compare:
- RequestBrief
- ArchitectureContextPacket
- BehaviorModel
- PseudocodePacket
- TestStrategyPacket
- ImplementationReport
- TestImplementationReport
- VerificationReport

The JudgeReport should decide whether:
- the implementation matches the PseudocodePacket
- the implementation satisfies the BehaviorModel
- the tests match the TestStrategyPacket
- the verification evidence is sufficient
- the scope remained bounded
- the architecture context was respected
- risks are honestly reported

Possible verdicts:
- PASS
- DESIGN_INCOMPLETE
- PSEUDOCODE_INCOMPLETE
- IMPLEMENTATION_MISMATCH
- TEST_COVERAGE_INCOMPLETE
- ARCHITECTURE_MISMATCH
- NEED_CONTEXT
- SCOPE_VIOLATION
- NEED_VERIFICATION
- BLOCKED

The judge should not rewrite the implementation.
The judge should identify the failed stage and the required correction.


4.10. Final report stage

Artifact produced:
FinalReport

Purpose:
Summarize the completed workflow for the user.

The FinalReport should include:
- original request
- workflow mode
- behavior designed
- architecture context used
- pseudocode summary
- tests designed
- implementation summary
- tests added or changed
- commands run
- verification result
- judge verdict
- unresolved risks
- follow-up recommendations if needed

The final report should be honest about missing verification or incomplete coverage.


5. Workflow modes

my-dev-kit-orchestrator should support five modes in v0.1.0:

- feature
- repair
- test
- refactor
- harden

Each mode reuses the same stage architecture but changes the entry point and some stage emphasis.


5.1. Feature mode

Command:
my-dev-kit-orchestrator start --mode feature "<request>"

Default command:
my-dev-kit-orchestrator start "<request>"

Purpose:
Build new or changed behavior accurately the first time.

Feature mode sequence:
1. RequestBrief
2. ArchitectureContextPacket
3. BehaviorModel
4. PseudocodePacket
5. TestStrategyPacket
6. ImplementationReport
7. TestImplementationReport
8. VerificationReport
9. JudgeReport
10. FinalReport

Feature mode should be the default.

Primary gate:
Implementation cannot begin until BehaviorModel and PseudocodePacket exist.

Testing gate:
Test implementation should be based on the TestStrategyPacket, not on ad hoc interpretation of the implementation.


5.2. Repair mode

Command:
my-dev-kit-orchestrator start --mode repair "<observed behavior>"

Purpose:
Repair behavior by comparing observed runtime behavior to intended design.

Repair mode is design-conformance repair.
It is not raw bug hunting.

Repair mode sequence:
1. ObservedBehaviorReport
2. ArchitectureContextPacket
3. BehaviorTrace
4. DivergenceReport
5. CorrectionDesign
6. RegressionTestStrategy
7. ImplementationReport
8. TestImplementationReport
9. VerificationReport
10. JudgeReport
11. FinalReport

Repair mode should first ask:
- what behavior was observed
- what behavior was expected
- where the expected behavior is defined
- what design or pseudocode path should control the behavior
- where actual runtime behavior first diverges from intended behavior

The divergence may be in:
- behavior model
- pseudocode packet
- implementation
- tests
- external contract
- runtime state ordering
- stale artifacts
- missing user expectation in the design

Primary gate:
Do not modify code until a divergence has been described and a correction design exists.


5.3. Test mode

Command:
my-dev-kit-orchestrator start --mode test "<test target>"

Purpose:
Design or implement comprehensive tests for existing behavior.

Test mode sequence:
1. TestTargetBrief
2. ArchitectureContextPacket
3. BehaviorReconstruction or BehaviorModel
4. PseudocodeSummary or PseudocodePacket
5. TestStrategyPacket
6. TestImplementationReport
7. VerificationReport
8. JudgeReport
9. FinalReport

Test mode should reconstruct the behavior before writing tests.

Primary gate:
Do not write test files until the behavior-derived TestStrategyPacket exists, unless the user explicitly asks for a small direct test patch.

Test mode should avoid turning into implementation unless a blocker is found and reported.


5.4. Refactor mode

Command:
my-dev-kit-orchestrator start --mode refactor "<refactor goal>"

Purpose:
Change code structure while preserving behavior.

Refactor mode sequence:
1. RefactorBrief
2. ArchitectureContextPacket
3. ExistingBehaviorMap
4. PreservedInvariantList
5. CompatibilityTestStrategy
6. RefactorPseudocodePacket
7. ImplementationReport
8. TestImplementationReport
9. VerificationReport
10. JudgeReport
11. FinalReport

Refactor mode should identify:
- behavior that must remain unchanged
- invariants that must be preserved
- current tests that protect behavior
- missing compatibility tests
- target structure
- files allowed to change
- verification commands

Primary gate:
Do not refactor until preserved behavior and compatibility tests are identified.


5.5. Harden mode

Command:
my-dev-kit-orchestrator start --mode harden "<hardening goal>"

Purpose:
Improve robustness, validation, and failure handling.

Harden mode sequence:
1. HardeningBrief
2. ArchitectureContextPacket
3. AssumptionReport
4. FailureModeMatrix
5. GuardPseudocodePacket
6. ResilienceTestStrategy
7. ImplementationReport
8. TestImplementationReport
9. VerificationReport
10. JudgeReport
11. FinalReport

Harden mode should identify:
- assumptions in current code
- what happens if each assumption is false
- relevant failure modes
- intended handling strategy for each failure mode
- guards or error states
- resilience tests
- remaining risks

Failure handling decisions may include:
- prevent
- recover
- surface to user
- log
- retry
- ignore safely
- fail fast

Primary gate:
Do not add guards or error handling until assumptions and failure modes are identified.


6. Stage gates

The workflow should rely on stage gates rather than long prompts.

A stage gate is a rule that prevents premature movement to the next kind of work.

Required gates:

Gate 1:
Do not design pseudocode until the BehaviorModel exists.

Gate 2:
Do not implement production code until the PseudocodePacket exists.

Gate 3:
Do not write test files until the TestStrategyPacket exists, unless the workflow explicitly allows a small direct test patch.

Gate 4:
Do not claim completion until VerificationReport exists.

Gate 5:
Do not mark a run as passed until JudgeReport returns PASS.

Gate 6:
Do not broaden scope without reporting a blocker or updating the relevant upstream artifact.

Gate 7:
Do not treat missing context as permission to guess. Report the missing context or return to the architecture context stage.

Gates should be enforced by:
- generated prompt stop conditions
- run status
- expected artifact names
- status output
- judge checks


7. Artifact handoff model

Artifacts are the main handoff mechanism.

The CLI should not rely on the coding agent remembering prior prompt text.

Each stage consumes previous artifacts and produces a new artifact.

Generic handoff chain:

RequestBrief
→ ArchitectureContextPacket
→ BehaviorModel
→ PseudocodePacket
→ TestStrategyPacket
→ ImplementationReport
→ TestImplementationReport
→ VerificationReport
→ JudgeReport
→ FinalReport

The next stage prompt should include:
- the names of required prior artifacts
- compact summaries or references to those artifacts
- the current task
- the required output artifact

The full prior artifacts should live in the run folder.

The prompt should not restate the whole workflow unless required.


8. Graph-guided retrieval integration

The architecture context stage should integrate with my-dev-kit.

In v0.1.0, this may be prompt-driven rather than automated.

The architecture context prompt should instruct the coding agent to use my-dev-kit when available to:
- index or refresh project artifacts if necessary
- search for relevant files and symbols
- retrieve graph-guided context
- inspect nearby dependencies
- identify existing tests
- identify docs or architecture references
- avoid broad whole-file reading until targeted retrieval is insufficient

my-dev-kit-orchestrator does not replace my-dev-kit.

The relationship is:
my-dev-kit provides project understanding.
my-dev-kit-orchestrator controls the design-to-code workflow.

Later versions may automate my-dev-kit execution directly.
v0.1.0 only needs to generate prompts that tell the coding agent how to perform the context retrieval stage.


9. Prompt architecture

Generated prompts should be stage-specific.

A stage prompt should include only:

1. Stage name
2. Workflow mode
3. Current run context
4. Required inputs
5. Task for this stage
6. Required output artifact
7. Stop conditions
8. Where to save or paste the output

Prompt design rules:
- no giant master prompt
- no unrelated workflow modes
- no reusable concrete scenario templates
- no excessive rule lists
- no hidden stage skipping
- no implementation instructions before implementation stage
- no test-file instructions before test implementation stage
- no success claim without verification evidence

Prompt length should be bounded.
The prompt should be long enough to define the stage clearly but not so long that it dilutes attention.

The full workflow is stored in project documents and run metadata.
The coding agent receives only the current stage instruction.


10. Run folder architecture

Default workspace:
.my-dev-kit-orchestrator/

Default run folder:
.my-dev-kit-orchestrator/runs/<run-id>/

Recommended structure:

.my-dev-kit-orchestrator/

  runs/
    <run-id>/
      00-request.txt
      run.json
      prompts/
      artifacts/

  07-test-implementation.prompt.txt

Artifact files:

Mode-specific artifact names may be used when appropriate:

Structured JSON schemas can come later.


11. Current stage selection

The command:

my-dev-kit-orchestrator prompt

should print the next prompt for the active run.

The next prompt is determined by:
- selected run
- workflow mode
- stage order
- which artifacts already exist
- whether the previous stage is blocked

Simple rule for v0.1.0:
The next stage is the first stage in the workflow sequence whose expected artifact is missing.

If a stage is marked blocked in run.json, status should show the block and prompt should print the blocked-stage recovery prompt or ask the user to resolve the blocker.

The command:

my-dev-kit-orchestrator prompt <stage>

prints a specific stage prompt.


12. Workflow correction and loop behavior

v0.1.0 does not need complex automatic looping.

However, the architecture should allow a judge to send the run back to a specific stage.

Possible judge outcomes and routing:

PASS
→ final report

NEED_CONTEXT
→ architecture context stage

DESIGN_INCOMPLETE
→ behavior model stage

PSEUDOCODE_INCOMPLETE
→ pseudocode packet stage

IMPLEMENTATION_MISMATCH
→ implementation stage

TEST_COVERAGE_INCOMPLETE
→ test strategy or test implementation stage

ARCHITECTURE_MISMATCH
→ architecture context or pseudocode packet stage

SCOPE_VIOLATION
→ stop and report

NEED_VERIFICATION
→ verification stage

BLOCKED
→ stop and report blocker

Routing does not need to be fully automated in v0.1.0.
The judge report can state the recommended next stage.


13. Design map direction

The long-term architecture should support a design map.

The design map links:

feature intent
→ behavior references
→ pseudocode steps
→ implementation symbols
→ tests
→ runtime observations

This design map supports both:
- writing code accurately the first time
- repairing behavior by tracing observed runtime behavior back to intended design

v0.1.0 does not need to fully implement the design map.
v0.1.0 should preserve artifact names and stage outputs in a way that allows design-map generation later.


14. Scope of v0.1.0

v0.1.0 should implement the workflow shell, not full automation.

Required:
- init command
- start command
- status command
- prompt command
- list command
- run folder creation
- mode selection
- stage prompt generation
- expected artifact tracking
- simple next-stage detection
- text artifact support
- feature, repair, test, refactor, and harden workflow definitions

Not required:
- direct LLM API execution
- full JSON schema validation
- full my-dev-kit command automation
- automatic artifact parsing
- automatic judge routing
- complex multi-agent execution
- full design-map generation
- CI integration

The first version should be useful even if every prompt is manually pasted into a coding agent.


15. Architectural success criteria

The workflow architecture is successful if:

- the user can start a workflow with one command
- the user can print the next prompt with one command
- the coding agent receives only one bounded stage at a time
- the workflow prevents premature implementation
- pseudocode exists before code implementation
- test strategy exists before test implementation
- implementation and tests share the same design source
- the run is inspectable and resumable
- verification evidence is separated from claims
- the final report states what was done and what remains uncertain

The tool should make the disciplined workflow easier to follow than an unstructured coding-agent prompt.
