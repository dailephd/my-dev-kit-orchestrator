AGENTS.txt

Main coding-agent operating manual for my-dev-kit-orchestrator.

Project identity

my-dev-kit-orchestrator is a CLI-first workflow tool for design-first software development with coding agents.

The project should help a user guide coding agents through a staged workflow:

request
-> architecture/context retrieval
-> behavior model
-> pseudocode packet
-> behavior-derived test strategy
-> implementation
-> test implementation
-> verification
-> judge
-> final report

The goal is not to create a large autonomous agent platform in the first version.

The goal is to make disciplined design-first coding easier than writing one large unstructured coding-agent prompt.

Primary purpose

Help coding agents write accurate code the first time by preventing premature implementation.

The coding agent should not jump directly from a vague request to code.

Before implementation, the workflow should produce:
- a clear request brief
- current project context
- a behavior model
- implementation-neutral pseudocode
- behavior-derived testing strategy

The implementation agent and testing agent should both use the same behavior model and pseudocode packet.

Core project direction

my-dev-kit-orchestrator is a separate project from:
- my-dev-kit
- my-dev-kit-lab

Relationship:

my-dev-kit
= code indexing, symbol lookup, graph-guided retrieval, and project understanding

my-dev-kit-orchestrator
= workflow orchestration, prompt generation, artifact handoff, and design-to-code staging

my-dev-kit-lab
= security validation, adversarial checks, and package-safety work for my-dev-kit

Do not merge my-dev-kit-orchestrator into my-dev-kit-lab.

This project is not primarily a security validation framework.

This project is not a debugging-only framework.

Repair is one supported workflow mode, but the central purpose is first-pass correctness.

Critical rules

1. Do not design this project as one giant prompt.

2. Do not create many unnecessary prompt templates if a simpler guideline-driven system is enough.

3. Do not expose every internal stage as a top-level CLI command.

4. Keep the command surface small and explicit.

5. Use the explicit command name:
   my-dev-kit-orchestrator

6. Do not use unclear abbreviations for the CLI name.

7. The CLI should behave like a workflow launcher and prompt manager.

8. The CLI should create runs, generate prompts, track artifacts, and identify the next stage.

9. The CLI does not need to call an LLM directly in v0.1.0.

10. The CLI does not need full schema validation in v0.1.0.

11. Use text artifacts in v0.1.0.

12. Keep workflow stages inspectable and resumable from files on disk.

13. Do not include reusable concrete scenario details inside generic prompts or guidelines.

14. Do not make the coding agent focus on examples instead of the current task.

15. Do not overconstrain the coding agent with excessive instructions.

16. Use strictness only for stage boundaries, required artifacts, scope control, and verification evidence.

17. Keep reasoning inside each stage flexible enough to adapt to the current project.

18. If required context is missing, report the missing context instead of guessing.

19. Do not claim code works unless verification evidence exists.

20. Do not claim tests passed unless they were actually run.

The first version should support:

my-dev-kit-orchestrator init
my-dev-kit-orchestrator start "<request>"
my-dev-kit-orchestrator start --mode feature "<request>"
my-dev-kit-orchestrator start --mode repair "<observed behavior>"
my-dev-kit-orchestrator start --mode test "<test target>"
my-dev-kit-orchestrator start --mode refactor "<refactor goal>"

my-dev-kit-orchestrator status
my-dev-kit-orchestrator prompt
my-dev-kit-orchestrator prompt <stage>
my-dev-kit-orchestrator list

Do not add many low-level commands in v0.1.0.

Avoid commands such as:
- my-dev-kit-orchestrator agent run
- my-dev-kit-orchestrator workflow execute
- my-dev-kit-orchestrator schema validate
- my-dev-kit-orchestrator artifact inspect
- my-dev-kit-orchestrator graph build
- my-dev-kit-orchestrator judge run

These may be considered later only if the basic workflow proves useful.

Workflow modes

Supported v0.1.0 modes:

1. feature

For building or changing behavior.

Sequence:
request brief
-> architecture context
-> behavior model
-> pseudocode packet
-> behavior-derived test strategy
-> implementation
-> test implementation
-> verification
-> judge
-> final report

2. repair

For observed behavior that diverges from intended behavior.

This is design-conformance repair, not raw bug hunting.

Sequence:
observed behavior report
-> architecture context
-> behavior trace
-> divergence report
-> correction design
-> regression test strategy
-> implementation
-> test implementation
-> verification
-> judge
-> final report

Repair mode should first connect observed behavior to intended behavior or pseudocode when those artifacts exist.

3. test

For designing or implementing comprehensive tests for existing behavior.

Sequence:
test target brief
-> architecture context
-> behavior reconstruction or behavior model
-> pseudocode summary or pseudocode packet
-> behavior-derived test strategy
-> test implementation
-> verification
-> judge
-> final report

4. refactor

For changing code structure while preserving behavior.

Sequence:
refactor brief
-> architecture context
-> existing behavior map
-> preserved invariant list
-> compatibility test strategy
-> refactor pseudocode packet
-> implementation
-> test implementation
-> verification
-> judge
-> final report

5. harden

For strengthening validation, guards, resilience, and failure handling.

Sequence:
hardening brief
-> architecture context
-> assumption report
-> failure-mode matrix
-> guard pseudocode packet
-> resilience test strategy
-> implementation
-> test implementation
-> verification
-> judge
-> final report

Stage gates

Use gates instead of giant prompts.

Gate 1:
Do not produce pseudocode until the behavior model exists.

Gate 2:
Do not implement production code until the pseudocode packet exists.

Gate 3:
Do not write test files until the test strategy packet exists, unless the task explicitly allows a small direct test patch.

Gate 4:
Do not claim completion until verification evidence exists.

Gate 5:
Do not mark a run as passed unless the judge report supports PASS.

Gate 6:
Do not broaden scope without reporting a blocker or updating the relevant upstream artifact.

Gate 7:
Do not treat missing context as permission to guess.

Artifacts

Core artifacts:

RequestBrief
ArchitectureContextPacket
BehaviorModel
PseudocodePacket
TestStrategyPacket
ImplementationReport
TestImplementationReport
VerificationReport
JudgeReport
FinalReport

Artifacts are the handoff mechanism between stages.

The coding agent should not rely on memory of previous prompts.

Each stage should consume the required prior artifacts and produce one named artifact.

Preferred filenames:

artifacts/request-brief.txt
artifacts/architecture-context-packet.txt
artifacts/behavior-model.txt
artifacts/pseudocode-packet.txt
artifacts/test-strategy-packet.txt
artifacts/implementation-report.txt
artifacts/test-implementation-report.txt
artifacts/verification-report.txt
artifacts/judge-report.txt
artifacts/final-report.txt

The pseudocode packet is the shared source of truth for implementation and testing.

The test strategy packet is the source of truth for test implementation.

Prompt generation rules

Do not generate a giant master prompt.

Generate one bounded prompt for the current stage.

A stage prompt should include:
- stage name
- workflow mode
- project root
- run folder
- required input artifacts
- task for this stage
- required output artifact
- stop conditions
- missing-context rule

A stage prompt should not include:
- unrelated workflow modes
- full project documentation
- every artifact contract
- reusable concrete scenario details
- instructions for future stages
- implementation instructions before the implementation stage
- test implementation instructions before the test implementation stage
- success claims before verification

Prompt length should be bounded.

The coding agent should receive the smallest useful instruction for the current stage.

Generic coding-agent guidance

When executing a stage:

1. Read the current stage prompt.

2. Read only the required input artifacts for that stage.

3. Use current project context, not assumptions from unrelated projects.

4. Produce the required output artifact.

5. Respect the stop conditions.

6. Report missing context instead of guessing.

7. Do not perform work from later stages.

8. Do not silently contradict prior artifacts.

9. If a prior artifact is wrong or insufficient, report the issue and recommend returning to the relevant stage.

10. Keep outputs specific to the current request and current repository.

Use of my-dev-kit

my-dev-kit is useful for architecture/context retrieval.

Use my-dev-kit in the architecture context stage when available.

Do not inject a full my-dev-kit command guide into every prompt.

Later stages should consume the ArchitectureContextPacket instead of repeating retrieval instructions.

Use the configured my-dev-kit access path.

If the project specifies a local my-dev-kit path, use that.

If the npm package is configured, use npm package execution syntax rather than assuming a global command.

Preferred package-style syntax when applicable:

npx @dailephd/my-dev-kit ...

Do not assume a bare global command exists.

Architecture context stage should use my-dev-kit to identify:
- relevant files
- relevant symbols
- relevant components or modules
- relevant routes, commands, services, or data boundaries
- relevant tests
- relevant docs
- state owners
- data owners
- upstream dependencies
- downstream consumers
- existing project patterns to preserve
- likely files or modules involved
- context gaps or uncertainty

Use focused retrieval before broad file reading.

Full-file read rule

Do not default to full-file reads.

If a full-file read is necessary, report:
- file path
- targeted searches attempted
- symbol lookups attempted
- slices attempted
- source or line-range reads attempted
- why targeted retrieval was insufficient
- what the broad read clarified

The goal is to understand the current architecture without flooding the agent with unnecessary code.

Behavior modeling

Before pseudocode or implementation, define the behavior.

The behavior model should identify:
- externally visible behavior
- internal supporting behavior
- state variables
- inputs
- events
- derived values
- invariants
- valid states
- invalid states
- boundaries and partitions
- empty states
- error states
- loading or pending states when relevant
- external contracts
- state transitions
- behavior to preserve
- behavior intentionally changed
- unresolved design questions

Do not write production code in the behavior model stage.

Do not write test files in the behavior model stage.

Pseudocode packet

The pseudocode packet converts the behavior model into implementation-neutral design.

The pseudocode packet should define:
- data flow
- state flow
- state transitions
- derived-value rules
- validation rules
- empty-state handling
- error-state handling
- external contract handling
- component, module, service, route, command, or helper contracts
- acceptance criteria
- likely files or modules to modify
- implementation constraints
- assumptions preserved
- assumptions removed or guarded
- unresolved implementation questions

Do not write production code in the pseudocode packet stage.

Do not write test files in the pseudocode packet stage.

The pseudocode packet should be precise enough to guide implementation and testing, but it should not contain full production patches.

Behavior-derived test design

Tests should be derived from the behavior model and pseudocode packet.

The test strategy should identify:
- behavior under test
- participating layers
- state variables
- events
- derived values
- invariants
- boundaries and partitions
- external contracts
- state transitions when relevant
- relevant failure modes
- existing tests
- test matrix
- test level assignment
- required verification commands
- coverage gaps
- risks not covered and why

Each test responsibility must trace to at least one of:
- behavior
- invariant
- state transition
- derived value
- boundary
- external contract
- failure mode

Use the narrowest useful test level:
- unit
- component
- integration
- end-to-end

Do not write test files in the test strategy stage unless explicitly requested.

Do not add test responsibilities that cannot be traced to the design artifacts.

Testing expectations

For meaningful code changes:
- design tests before writing test files
- use existing project test conventions
- prefer narrow tests for deterministic logic
- use integration tests for module, route, service, artifact, storage, CLI, or boundary behavior
- use end-to-end tests only for critical user-visible or externally observable workflows
- cover relevant boundaries and state transitions
- cover relevant external contracts
- cover relevant failure modes
- report uncovered risks honestly

Do not claim an edge case is covered unless a test or explicit runtime check proves it.

Implementation rules

During implementation:
- follow the ArchitectureContextPacket
- implement the PseudocodePacket
- preserve the BehaviorModel
- modify only justified files
- avoid parallel architecture
- avoid unrelated cleanup
- avoid broad rewrites
- preserve existing project patterns unless the pseudocode or refactor mode justifies a change
- report blockers instead of guessing
- document deviations from the pseudocode packet
- do not claim verification success without command evidence

If implementation reveals that the pseudocode is wrong or impossible, stop and report that the workflow should return to the pseudocode stage.

Test implementation rules

During test implementation:
- follow the TestStrategyPacket
- preserve traceability from tests to behavior
- prefer the narrowest useful test level
- report strategy items that cannot be implemented
- do not replace the TestStrategyPacket with a new unrelated strategy
- do not change production behavior unless reporting a blocker
- do not claim tests passed unless command evidence exists

Verification rules

After meaningful code or test changes:
- run the narrowest relevant checks first
- then broader validation if feasible
- record actual commands
- record working directories
- record exit codes
- summarize failures honestly
- state skipped checks and reasons
- never claim checks passed unless they were run

Possible checks:
- targeted unit tests
- targeted component tests
- targeted integration tests
- relevant end-to-end tests
- typecheck
- lint
- build
- smoke checks

Judge rules

The judge stage should compare actual work against prior artifacts.

The judge should review:
- RequestBrief
- ArchitectureContextPacket
- BehaviorModel
- PseudocodePacket
- TestStrategyPacket
- ImplementationReport
- TestImplementationReport
- VerificationReport

Judge questions:
- Does the implementation match the PseudocodePacket?
- Does the implementation satisfy the BehaviorModel?
- Do tests match the TestStrategyPacket?
- Were important behaviors, invariants, transitions, boundaries, contracts, and failure modes covered?
- Was the architecture context respected?
- Did the implementation stay in scope?
- Were verification commands actually run?
- Are risks and gaps reported honestly?

Possible verdicts:
- PASS
- DESIGN_INCOMPLETE
- PSEUDOCODE_INCOMPLETE
- IMPLEMENTATION_MISMATCH
- TEST_COVERAGE_INCOMPLETE
- ARCHITECTURE_MISMATCH
- NEED_CONTEXT
- SCOPE_VIOLATION
- NEED_VERIFICATION
- BLOCKED

Do not approve without verification evidence.

Do not hide uncertainty.

Repair mode guidance

Repair mode should not begin with unfocused code hunting.

Repair mode should begin by connecting observed behavior to intended behavior.

The repair workflow should identify:
- observed behavior
- expected behavior
- relevant design artifact if available
- relevant pseudocode path if available
- actual runtime path if available
- first divergence point
- likely correction category

Correction category may be:
- behavior model update
- pseudocode update
- implementation correction
- test correction
- external contract correction
- runtime state ordering correction
- artifact update

Do not modify code until the divergence has been described and a correction design exists.

Refactor mode guidance

Refactor mode must preserve behavior.

Before refactoring:
- identify existing behavior
- identify preserved invariants
- identify current tests
- identify missing compatibility tests
- define refactor pseudocode
- define verification commands

Do not refactor until preserved behavior and compatibility tests are identified.

Harden mode guidance

Hardening mode should identify assumptions and failure modes before adding guards.

Before implementation:
- identify assumptions
- state what breaks if each assumption is false
- identify relevant failure modes
- decide how each failure should be handled
- define guard or error-state pseudocode
- define resilience tests

Failure handling decisions may include:
- prevent
- recover
- surface to user
- log
- retry
- ignore safely
- fail fast

Do not hide architectural problems behind fallback behavior.

Do not silently swallow errors.

Preserve useful diagnostics.

Final report expectations

Every completed run should produce a final report with:
- original request
- workflow mode
- run ID
- stages completed
- architecture context used
- behavior designed
- pseudocode summary
- test strategy summary
- implementation summary
- tests added or updated
- verification commands run
- verification results
- judge verdict
- unresolved risks
- follow-up recommendations if needed

Do not exaggerate success.

Do not omit failed or skipped verification.

Do not hide remaining risks.

Documentation-only changes

For documentation-only changes:
- do not run implementation workflow unnecessarily
- inspect relevant docs
- update only documentation files
- run git status
- run available markdown lint only if the repo provides it
- do not claim production behavior changed
- verify no production code changed

When uncertain

If uncertain:
- stop and state the uncertainty
- explain what information is missing
- explain which stage should be revisited
- do not silently broaden scope
- do not replace architecture without evidence
- do not invent behavior not supported by artifacts
